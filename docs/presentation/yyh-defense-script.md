# yyh 负责部分答辩讲稿

## 第 1 页：Web 遥控设计

这一页我主要介绍小车遥控部分是怎么从 Web 页面连到真实小车的。

我们没有让浏览器直接连接小车，而是在中间加了一层 FastAPI 后端。页面上的前进、后退、左转、右转、停止和急停按钮，都会先调用后端的 `/api/control/manual` 接口。后端再根据当前选择的小车控制端口，把指令编码成小车 Rosmaster/App 能识别的 TCP 控制帧，然后发送到小车。

这套设计的好处是浏览器端比较简单，只负责用户交互；真正和小车通信、做异常保护、判断端口、处理超时的逻辑都放在后端。比如我们现在优先使用小车原生的 `6000` 控制端口，如果 `6000` 不可用，也可以切换到自建的 `6001` 桥接服务。

遥控部分还加入了安全逻辑。前进和后退支持长按连续运动，方便手动移动小车；普通遥控页的左转和右转改成短脉冲，最多大约 0.5 秒，防止小车持续旋转造成危险。急停按钮是最高优先级，一旦触发，会直接抢占当前控制状态，连续发送停止帧，避免前面排队的动作继续执行。

如果老师问为什么不让前端直接连小车，可以回答：第一，浏览器直接连 TCP 服务不方便，也不安全；第二，后端可以统一做协议转换、端口切换、超时降级和急停抢占；第三，后续语音控制、手势控制、巡航控制都能复用同一个后端控制接口。

## 第 2 页：SLAM 建图与自动导航

这一页是我负责部分里最核心的一页，需要重点讲。

SLAM 的全称是 Simultaneous Localization and Mapping，也就是“同步定位与建图”。简单来说，小车一边移动，一边利用激光雷达扫描周围环境，同时结合底盘里程计估计自己的运动轨迹，逐步生成一张二维栅格地图。地图里通常白色代表可通行区域，黑色代表墙体或障碍物，灰色代表未知区域。

我们把小车原本比较复杂的 ROS2 / Docker / 图形化操作流程封装到了 Web 平台里。用户在页面上点击“开始建图”，后端会通过 SSH 连接小车，并在小车的 ROS2 Docker 容器里启动建图 launch。当前页面默认使用 `gmapping` 建图，后端也预留了 `cartographer` 和 `rtabmap` 的接口。建图过程中，用户可以用页面右侧的遥控器慢速移动小车，让激光雷达扫完整个房间。

建图完成后，点击“保存地图”，系统会把地图保存为 `.yaml` 和 `.pgm` 文件。`.pgm` 是栅格图像，`.yaml` 保存地图分辨率、原点坐标、图片路径等元数据。我们后来也修复了保存地图后下拉框不刷新的问题：后端会同步检查 Docker 容器内和小车宿主机 maps 目录，确保新地图能够出现在 Web 下拉框里。

导航阶段和建图阶段不同。导航不是继续画图，而是加载已经保存好的地图，并启动 Navigation2。导航页面可以选择 `DWA`、`TEB` 和 `RTABMap` 三种导航方式。实际演示时默认推荐 `DWA`，它更适合低速移动和现场稳定演示；`TEB` 更强调轨迹时间优化和动态约束；`RTABMap` 适合更复杂的视觉/图优化场景，但实训演示链路更重。

导航前必须先设置小车在地图中的当前位置。这一步对应 ROS2 里的 `/initialpose`。原因是：保存下来的地图只是一个坐标系，小车重新开机或重新启动导航后，系统并不知道小车现在处在旧地图的哪个位置。用户需要在 Web 地图上点选小车真实位置和朝向，后端发布初始位姿给 AMCL。AMCL 接收到初始位姿后，会结合激光雷达实时扫描和保存地图进行匹配，逐渐估计出 `map -> base_footprint` 的变换。只有 AMCL 定位成功后，发送目标点才可靠。

目标点发送以后，后端把 Web 上点选的坐标转换成 ROS2 的目标位姿，发布给导航系统。Nav2 会根据地图、代价地图和当前位姿规划路径，再由控制器输出速度指令，小车就能自动移动到目标点。

### 第 2 页流程图讲法

PPT 里的流程图建议按泳道讲：

- 用户泳道：检查连接、开始建图、遥控小车扫图、保存地图、选择地图、设置当前位置、点选目标点。
- Web 前端泳道：展示地图、采集按钮动作、采集点选坐标、显示运行状态。
- FastAPI 后端泳道：调用控制接口、通过 SSH 启动 ROS2 launch、保存/同步地图、发布 `/initialpose` 和目标点。
- 小车 ROS2 泳道：激光雷达扫描、gmapping 建图、map_server 加载地图、AMCL 定位、Nav2 规划和控制底盘。

讲的时候强调一点：建图阶段解决“环境是什么样”，导航阶段解决“我在地图哪里、怎么去目标点”。AMCL 不回传当前位置时，不一定是程序坏了，很多时候是因为还没有进入导航模式，或者没有设置初始位姿。

## 第 3 页：夜间定点巡航设计

第三页讲的是我们结合智能家居场景设计的夜间自动巡航。

需求背景是：晚上主人和屋内人员睡觉后，小车可以按预设路线在房间里低速巡航，用于安全巡视、异常提醒或者后续接入烟雾、火焰、人体检测等功能。这个功能不是简单写死一条路线，而是允许用户在 Web 页面上自己设计巡航线路。

具体使用流程是：用户先进入“巡航”页面，用页面上的精细遥控器控制小车移动。每到一个关键位置，比如客厅、门口、书房、卧室，就点击“保存途经点”。途经点可以命名、删除、调整顺序，也可以保存成一条巡航路线。下一次测试时可以直接加载已经保存好的路线。

路线规划部分使用的是 A* 网格搜索。我们把巡航页面上的环境抽象成二维网格，把途经点映射成网格坐标。A* 会在相邻途经点之间搜索一条代价较低的路径。这里我们的代价不只考虑距离，还加入了“转弯惩罚”，也就是尽量减少频繁拐弯。因为小车在真实地面上靠短脉冲控制，转弯误差比直行更容易累积，所以少转弯比单纯最短格子数更适合现场演示。

巡航执行逻辑是：如果设置 A、B、C 三个途经点，小车会按 A -> B -> C 行驶，到达每个点后蜂鸣提示并停留设定秒数。到最后一个点后，不是随便绕一个闭环，而是原地掉头，再按 C -> B -> A 返回。这样路线更符合夜间来回巡视的场景，也更容易控制误差。页面支持只巡航一次、巡航指定次数、循环巡航三种模式。

需要说明的是，当前巡航功能更偏“上层路线设计和低速定点巡航”，它使用 Web 网格和脉冲控制估计小车位置；而 SLAM/Nav2 导航是小车 ROS2 系统基于激光雷达和地图实时定位、规划、避障。两者定位方式不同，应用场景也不同：SLAM 导航适合点到点自动导航；定点巡航适合预设夜间巡视路线和任务编排。

## 可能被老师追问的问题

### 1. A* 和 Dijkstra 有什么区别？

Dijkstra 会从起点开始向外扩展，系统地计算到各个点的最短代价，不依赖目标方向信息。它很稳，但在大地图里会探索很多和目标方向无关的节点。

A* 可以看成在 Dijkstra 的基础上加入了启发函数。它的评价函数一般是：

```text
f(n) = g(n) + h(n)
```

其中 `g(n)` 是从起点走到当前节点的实际代价，`h(n)` 是当前节点到目标点的估计代价。我们这里的 `h(n)` 可以理解为到目标点的曼哈顿距离。因为有了目标方向的引导，A* 通常比 Dijkstra 搜索更快，尤其适合 Web 页面这种网格路径规划。

如果启发函数设计得合理，比如不高估真实距离，A* 仍然可以保证找到最优路径。我们还在 `g(n)` 里加入转弯惩罚，让规划结果更适合小车实际运动。

### 2. A* 和小车原生 Nav2 导航有什么区别？

Nav2 是完整机器人导航系统，它依赖真实地图、AMCL 定位、代价地图、路径规划器和控制器，会根据雷达和地图实时调整运动。它更适合真实环境中的自主导航。

我们的 A* 巡航是 Web 层的路线编排和轻量规划，用来把几个途经点连成一条巡航路线，并生成前进、左转、右转这类动作序列。它不直接替代 Nav2，也不等同于实时避障。它的优势是简单、可解释、适合预设夜间巡航任务；不足是依赖人工标定网格和脉冲时间，真实环境中需要低速运行并保守设置。

答辩时可以这样总结：Nav2 解决“机器人在真实地图中自主导航”，A* 巡航解决“用户设计一条可重复执行的巡视路线”。它们不是互相替代，而是两个层级的功能。

### 3. 为什么巡航要尽量减少转弯？

因为小车用短脉冲控制时，直行误差相对更容易估计，而转弯误差会受地面摩擦、电量、速度、轮子打滑影响。频繁转弯会让实际位置和页面估计位置偏差越来越大。所以我们在 A* 代价里加入转弯惩罚，让路线尽量平直、少拐弯，更适合演示和夜间低速巡航。

### 4. 为什么导航前一定要确认当前位置？

保存地图以后，地图坐标系是固定的，但小车重新开机或重新启动导航后，系统不知道它在这张地图里的绝对位置。确认当前位置就是给 AMCL 一个初始估计。AMCL 才能把激光雷达扫描和地图匹配起来，生成小车当前位姿。如果没有这一步，Nav2 就会报 `map` 到 `base_footprint` 的 TF 不可用，或者目标发出去了但小车不动。

## 简短收尾

我负责的这几部分主要围绕“让真车能力可以被 Web 平台稳定调用”。遥控部分解决基础控制和安全急停；SLAM 与导航部分把复杂的 ROS2 操作封装成网页流程；夜间定点巡航则面向智能家居场景，提供可保存、可复用、可解释的巡视路线设计。
